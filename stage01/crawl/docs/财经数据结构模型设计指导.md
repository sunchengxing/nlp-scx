# 财经数据结构模型设计指导

## 一、前置思考：先搞清楚数据源和用途

在设计结构之前，先回答两个问题：

1. **数据从哪来？** — A股行情接口（如东方财富、新浪、Tushare、AKShare）返回的字段是什么？
2. **用来做什么？** — 只是存储展示？还是要做量化分析、策略回测？

> 做回测就需要**复权因子**，单纯展示就不需要。用途决定字段粒度。

---

## 二、模块拆分

按数据类型拆成 3 个模型文件：

| 文件 | 数据类型 | 核心实体 |
|------|---------|---------|
| `kline.py` | K线数据 | 日线/周线/月线/分钟线 |
| `dragon_tiger.py` | 龙虎榜 | 每日上榜个股详情 |
| `capital_flow.py` | 资金流向 | 主力/散户资金流入流出 |

所有文件放在 `core/models/` 目录下。

---

## 三、K线模型（kline.py）

### 核心问题：日线/周线/月线/分钟线要不要拆成4个类？

**建议：不拆，用 1 个类 + 1 个 `period` 枚举字段搞定。**

理由：
- 字段完全一致（开、高、低、收、量、额），只是时间粒度不同
- 拆成4个类会导致重复代码，后续改一个字段要改4处
- 通过 `period` 区分，查询时过滤即可

### 必备字段清单

| 字段 | 说明 | 为什么需要 |
|------|------|-----------|
| symbol | 股票代码 | 唯一标识 |
| date / datetime | 时间戳 | 日线用 date，分钟线需要精确到分钟 |
| open / high / low / close | OHLC | K线最基本四价 |
| volume | 成交量（手） | 量能分析 |
| amount | 成交额（元） | 量价配合分析 |
| turnover | 换手率 | 判断活跃度 |
| adj_factor | 复权因子 | **做回测必须，展示可选** |
| period | 周期类型 | 日/周/月/1min/5min/15min/30min/60min |

### 待决策点

- [ ] 分钟线是否需要单独一个类？—— 分钟线数据量巨大（1只股票1天240条1分钟线），存储策略和日线不同
- [ ] 是否需要 `pre_close`（昨收）？—— 计算涨跌幅用
- [ ] 时间字段用 `date` 还是 `datetime`？—— 日线及以上用 date，分钟线用 datetime，是否统一？

## 四、龙虎榜模型（dragon_tiger.py）

### 数据是两层结构

```
龙虎榜当日记录（1条）
  ├── 买入席位列表（N条）
  └── 卖出席位列表（N条）
```

**必须设计 2 个类：**

### 4.1 DragonTigerRecord — 当日个股上榜记录

| 字段 | 说明 |
|------|------|
| symbol | 股票代码 |
| date | 上榜日期 |
| reason | 上榜原因（如：日涨幅偏离值达7%） |
| deviation | 涨跌幅偏离值 |
| close_price | 当日收盘价 |
| change_pct | 当日涨跌幅 |

### 4.2 DragonTigerSeat — 席位明细

| 字段 | 说明 |
|------|------|
| record_id | 关联的上榜记录 ID |
| seat_name | 席位名称 |
| branch | 营业部名称 |
| buy_amount | 买入金额 |
| sell_amount | 卖出金额 |
| net_amount | 净买入金额 |
| side | 买方/卖方 |

### 待决策点

- [ ] `side` 用枚举（BUY/SELL）还是布尔字段？
- [ ] 是否需要 `is_inst`（是否机构席位）？—— 机构专用席位和游资营业部分析逻辑完全不同
- [ ] 席位和记录的关系：是用嵌套结构（record 内含 seats 列表）还是扁平结构（seat 通过 record_id 关联）？

## 五、资金流向模型（capital_flow.py）

### 数据本质：按金额大小给成交量分类

| 类别 | 常见定义 |
|------|---------|
| 超大单 | >100万 |
| 大单 | 20-100万 |
| 中单 | 4-20万 |
| 小单 | <4万 |

每个类别需要记录：**流入金额、流出金额、净流入**

### 字段设计

| 字段 | 说明 |
|------|------|
| symbol | 股票代码 |
| date | 日期 |
| huge_buy_in | 超大单流入 |
| huge_sell_out | 超大单流出 |
| huge_net | 超大单净流入 |
| big_buy_in | 大单流入 |
| big_sell_out | 大单流出 |
| big_net | 大单净流入 |
| mid_buy_in | 中单流入 |
| mid_sell_out | 中单流出 |
| mid_net | 中单净流入 |
| small_buy_in | 小单流入 |
| small_sell_out | 小单流出 |
| small_net | 小单净流入 |
| main_net_inflow | 主力净流入（超大单净 + 大单净） |
| total_amount | 总成交额（校验数据一致性用） |

### 待决策点

- [ ] `main_net_inflow` 作为冗余字段存储？还是每次查询时计算？—— 冗余字段方便查询但需保证一致性
- [ ] 分钟级资金流向要不要支持？—— 数据量指数级增长
- [ ] 四个类别的流入/流出/净流入，是用扁平字段还是嵌套结构？—— 扁平字段简单直接，嵌套结构更优雅但查询麻烦

## 六、共性设计建议

1. **所有模型都加 `source` 字段** — 记录数据来源（东方财富/新浪/Tushare），不同源数据可能有差异
2. **所有模型都加 `fetched_at` 字段** — 记录采集时间，便于排查数据问题
3. **用 Pydantic 而不是 dataclass** — 爬虫项目天然需要数据校验，Pydantic 能自动类型检查 + 导出 JSON Schema
4. **枚举用 `enum.Enum`** — period、side 这些不要用字符串硬编码
5. **统一 symbol 格式** — 决定用 `600000` 还是 `sh600000` 还是 `SH600000`，全局统一
